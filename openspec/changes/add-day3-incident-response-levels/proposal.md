# Proposal: Add Day 3 Incident Response Levels

## Why

平台目前有 12 个关卡（Chapter 1: 基础命令，Chapter 2: 权限实战）。
这两章的共同特点是：**单条命令解决一个问题**。

Chapter 3 需要质的跳跃——真实的 Linux 运维工作从不是单条命令，而是多个工具的组合。

用户将扮演**值班工程师**，在双11电商大促期间收到线上告警，通过一关一关地排查问题，逐步掌握：

- 进程管理与系统诊断（ps, top, ss, lsof, kill）
- 文本处理与过滤（grep, cut, awk, wc）
- 管道组合（将多个命令串联解决复杂问题）
- 模拟真实生产事故的完整排查流程

同时，Chapter 3 的引入也暴露了一个现有 UI 设计缺陷：
**"目标命令"区域直接展示完整命令**，对复杂管道命令会完全剥夺学习过程。
本提案同步引入 **知识卡片（Knowledge Card）** 功能，在界面上解释命令含义，
并将"目标命令"改造为"任务目标"（描述结果，而非给出答案）。

## What Changes

### 新增关卡：Chapter 3 - 事故响应（Level 13–20）

**叙事主线**：双11凌晨2:17，监控告警——网站响应超时，部分用户无法下单。

| Level | 标题 | 场景 | 核心命令 | 难度 |
|-------|------|------|----------|------|
| 13 | 第一响应 | CPU 飙到 99%，找出占用最高的进程名 | `ps aux --sort=-%cpu \| head` | ★★☆ |
| 14 | 磁盘告急 | 日志盘使用率 95%，找出哪个目录最大 | `du -sh /var/log/* \| sort -rh \| head` | ★★★ |
| 15 | 端口被占 | 重启服务失败，找出占用 8080 端口的进程 | `ss -tlnp \| grep 8080` | ★★★ |
| 16 | 日志追踪 | 从 nginx 日志找出所有 HTTP 500 错误 | `grep " 500 " access.log` | ★★☆ |
| 17 | 统计告警 | 老板问：今天 ERROR 发生了多少次？ | `grep "ERROR" app.log \| wc -l` | ★★★ |
| 18 | IP 追凶 | 找出访问量最大的 5 个 IP（可能是攻击者） | `awk '{print $1}' \| sort \| uniq -c \| sort -rn \| head -5` | ★★★★ |
| 19 | 时间取证 | 精确统计告警时段（02:17）内的 500 错误数 | `grep "02:17" \| grep " 500 " \| wc -l` | ★★★★ |
| 20 | 终极取证 | 找出告警时段内请求最多的 IP 及其请求次数 | 多管道组合 | ★★★★★ |

### 新增功能：关卡知识卡片（Knowledge Card）

每个关卡新增 `knowledgeCards` 字段，在界面上展示本关涉及的命令解释：
- 命令的作用
- 关键参数/标志的含义
- 管道符 `|` 的概念解释（首次出现时）

Level 数据结构新增字段（详见 design.md）。

### 调整：Level 界面"目标命令"→"任务目标"

将 `Level` 组件中展示完整命令的区域，改为描述任务结果的文字。
对已有的 Chapter 1/2 关卡保持向后兼容（仍可显示 `command` 字段）。

## Impact

- **用户影响**：通过真实事故排查场景，掌握 Linux 管道和文本处理能力
- **技术影响**：
  - 前端：扩展 Level 类型定义，添加 `knowledgeCards` 和 `objective` 字段；更新 Level 组件
  - 后端：扩展验证逻辑，支持输出内容验证（output_contains / output_matches）
  - Docker：容器预埋 nginx access.log（~5000 行，含植入的攻击 IP 和 500 错误）、app.log、模拟占用 CPU/端口的进程
- **兼容性**：完全向后兼容，Chapter 1/2 关卡无需修改
